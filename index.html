Building cinematic websites with Claude

Everything used to build the kind of site that makes people stop scrolling: the stack, the exact resources for each effect, the build order, and the prompts to actually get Claude to build it.

Expo Claude Code GSAP Three.js Framer Motion
~/setup $

Getting Started: Expo + Claude Code

Expo Router renders standard React to the web when you run it in web mode, which means every library in this guide, GSAP, Framer Motion, React Three Fiber, works exactly the way it would in any other React project. The routing is just file-based instead of manually configured.

Project setup

  1. Install the CLI: npx create-expo-app@latest
  2. Choose the blank TypeScript template for the cleanest starting point
  3. Run it in web mode: npx expo start --web
  4. Docs if you get stuck: docs.expo.dev
  5. Install your animation libraries as normal npm packages, nothing Expo-specific needed for any of them

Claude Code setup

  1. Install Claude Code: npm install -g @anthropic-ai/claude-code
  2. Run claude from inside your project root
  3. Docs if you get stuck: docs.claude.com/claude-code
  4. Give it context first: what the site is, who it's for, what effect you're building toward before asking for code
  5. Build one effect at a time, in the order laid out at the bottom of this guide, don't ask for everything in a single prompt

Hero Scroll-Pinned Video

The hero video is the first three seconds of trust. It should feel cinematic, pin in place while the page scrolls past it, and scrub or fade with scroll position instead of just autoplaying in the background.

Resources
Higgsfield
AI video generation built specifically for cinematic camera moves (dolly, pan, orbit). Best option right now for a hero clip that needs to feel directed, not generated.
Claude + MCP video tools
If you have an MCP video-capable connector enabled in Claude Code, you can generate, trim, and re-export hero footage directly inside your build session instead of round-tripping through a separate app.
Google Flow
Google's AI filmmaking tool. Strong at longer, more controllable camera paths, useful when a clip's length is too short for your pin distance.
Adobe Firefly Video
Free tier caps out around 2 generations, but the output quality is solid enough to use as your final hero clip if you nail the prompt on the first couple tries.
GSAP + ScrollTrigger
Pins the section and scrubs playback or opacity against scroll position.
npm install gsap
Lenis
Optional, but pairs well here if you want the scrub itself to feel eased rather than tied 1:1 to raw scroll.
npm install lenis
How to build it
  1. 01Generate or source your hero video (see resources above). Export at the resolution you'll actually display, don't ship a 4K file for a 1200px-wide hero.
  2. 02Compress it. Run it through ffmpeg or Handbrake to get file size down, a hero video over 5-8MB will hurt your load time badly.
  3. 03Wrap the hero in a tall container (e.g. 250-300vh) with the video itself set to position: sticky or fixed inside it, this is what creates the 'pin' feeling.
  4. 04Install GSAP and ScrollTrigger, then scrub the video's currentTime (or a CSS scale/opacity value) against the ScrollTrigger progress so the video responds to scroll instead of just playing on its own.
  5. 05Add a poster image and preload="auto" so the first paint isn't a blank box while the video buffers.
  6. 06Test specifically on mobile Safari, background video autoplay needs playsinline muted autoplay or iOS will simply refuse to play it.
Prompt it
claude
I want to build a scroll-pinned cinematic hero video for my landing page, similar to what you see on high-end agency and product sites, where the video feels locked to the viewport and responds to how far the user has scrolled rather than just playing on a loop in the background. I'm open to how you approach the actual mechanics, GSAP with ScrollTrigger is my default assumption since it's the most battle-tested option for scrubbing video playback or transform values against scroll progress, but if you think Framer Motion's useScroll and useTransform would produce a cleaner result given the rest of my component structure, use that instead. Feel free to layer in Lenis for the scroll physics if you think the pin will feel more premium eased rather than tracking raw scroll 1:1. I care about the following things being genuinely good, not just functional: the release from the pin into the next section should have zero visual jump, the video should have a real poster fallback and sensible preload behavior so there's no blank flash on first load, and it needs to actually autoplay correctly on mobile Safari, which has its own quirks around muted/playsinline video that trip a lot of implementations up. If there's a smarter or more modern approach than what I've described, for example using the native View Transitions API for part of this, or handling the scrub via a custom hook rather than raw ScrollTrigger callbacks, go with your best judgment rather than sticking rigidly to what I've outlined. Optimize for it feeling expensive and intentional over it being the simplest possible implementation.
~/build/02-cursor-fx $

Cursor-Based Animations

A custom cursor that lags slightly behind the real one, and buttons that pull the cursor toward their center, this is one of the cheapest ways to make a site feel expensive.

Resources
Framer Motion
useMotionValue plus a spring gives you a cursor-follow element with almost no code.
npm install framer-motion
GSAP quickTo()
Purpose-built for lerping a value toward a target every frame with minimal overhead.
npm install gsap
Cursify
An open-source library of prebuilt custom cursor effects if you want a starting point rather than building the lerp logic from scratch.
npm install cursify
Rive
Overkill for a simple follow-cursor, but excellent if you want the cursor itself to be a small animated vector character that reacts to hover states.
How to build it
  1. 01Render a single fixed-position div as your custom cursor, hide the real OS cursor with cursor: none on desktop only.
  2. 02Track mousemove coordinates in a ref, don't trigger a React re-render on every mousemove, update the DOM element directly for performance.
  3. 03Use GSAP's quickTo() or a Framer Motion spring to ease the custom cursor's position toward the real cursor's coordinates, this lag is what makes it feel intentional instead of glitchy.
  4. 04For magnetic buttons, calculate the distance from the cursor to the button's center on mousemove, and nudge the button toward the cursor when it's within range.
  5. 05Explicitly disable all of this on touch devices, check for a touch capable pointer and skip the custom cursor entirely rather than leaving a ghost cursor on mobile.
Prompt it
claude
I want a custom cursor system across my whole site, not just a single follow-the-mouse dot, but something that feels like a real piece of interface design: the base cursor should lag slightly behind the real pointer with its own easing personality, and it should be able to morph, resize, or change label depending on what it's hovering (bigger over images, showing a small 'view' or 'drag' label over specific elements, shrinking over text). I'd lean toward GSAP's quickTo for the core lerp since it tends to be the smoothest option for this kind of continuous per-frame update, but I'm open to Framer Motion springs if you think the rest of my animation stack makes that a better fit, and feel free to look at Cursify for inspiration or as a base if it saves meaningful time over hand-rolling everything. On top of the base follow behavior, add a magnetic pull on primary buttons and nav links, where the button itself (or its inner label) shifts toward the cursor within a reasonable proximity radius and springs back cleanly on mouse leave, use your judgment on the exact radius and pull strength rather than a fixed number, tune it until it feels good rather than mechanical. Make sure there's zero trace of any of this on touch devices, and think about whether the cursor should also respond to scroll velocity or drag states somewhere on the site, if there's a natural place for that kind of extra polish, add it.
~/build/03-scroll-reveal $

Scroll-Triggered Reveals

Content that fades or slides in as it enters the viewport, done well this reads as polish, done with default easing on every single element it reads as a template.

Resources
GSAP ScrollTrigger
Full control over trigger points, scrub vs. one-shot, and stagger timing across grouped elements.
npm install gsap
Framer Motion whileInView
A couple lines per component, good default choice if you don't need scrubbing.
npm install framer-motion
Intersection Observer (vanilla)
Zero dependencies if you're not already using GSAP or Framer Motion elsewhere.
AOS (Animate On Scroll)
A tiny, attribute-based library if you want reveals without writing any JS logic at all.
npm install aos
How to build it
  1. 01Decide once, upfront, on a single easing curve and duration for every reveal on the site, consistency here matters more than any individual animation being clever.
  2. 02Group related elements (e.g. a row of 3 cards) and stagger their reveal by 60-100ms each, rather than animating them all at the exact same instant.
  3. 03Trigger the reveal when the element is about 15-20% into the viewport, not exactly at the very edge, this avoids a jarring 'just appeared' pop.
  4. 04Animate opacity and a small transform together, never just opacity alone, the subtle movement is what reads as intentional.
  5. 05Set will-change on elements right before they animate and remove it after, to keep scroll performance smooth on lower-end devices.
Prompt it
claude
Design a scroll reveal system for the whole site, not just a single one-off animation on one section, something consistent enough that every content block feels like it belongs to the same design language, but with enough variation that it doesn't feel mechanically identical everywhere. Use whichever combination of GSAP ScrollTrigger, Framer Motion's whileInView, or the raw Intersection Observer API makes the most sense given how the rest of the components are structured, and feel free to mix approaches if certain sections genuinely call for scrubbed animation (tied continuously to scroll position) while others are better as a one-shot reveal that fires once and stays. Think carefully about stagger timing for grouped elements like card grids or feature lists, sequential reveals almost always look better than everything appearing simultaneously, but don't default to a single generic fade-up for every element type, headlines, images, and body copy can each have a treatment suited to what they are. I want the timing, easing curves, and trigger thresholds to feel considered rather than left at library defaults, use your own judgment on the specifics rather than a fixed recipe, and prioritize this feeling refined over it being the fastest thing to implement.

Horizontal Scroll / Scrollytelling

A section that pins in place while normal vertical scrolling drives horizontal movement, common in portfolio and case-study sections where you want a gallery-like feel.

Resources
GSAP ScrollTrigger (horizontal pin)
The standard approach, pin the section and translateX its inner track based on scroll progress.
npm install gsap
Locomotive Scroll
Handles this natively with data attributes, a good option if you want horizontal scroll sections without hand-writing the pin math.
npm install locomotive-scroll
Fullpage.js
Built for full-section, slide-like navigation, better suited if your whole page is structured this way.
npm install fullpage.js
React Three Fiber
Worth considering if the horizontal track should live in an actual 3D scene rather than a flat DOM row.
npm install three @react-three/fiber @react-three/drei
How to build it
  1. 01Build the horizontal content as a single flex row wider than the viewport, don't rely on the browser's native horizontal scrollbar.
  2. 02Wrap that row in a pinned section sized to however much vertical scroll distance you want the horizontal movement to take.
  3. 03Use ScrollTrigger to translateX the track from 0 to -(track width - viewport width) as the user scrolls through the pin.
  4. 04Add horizontal-scroll-aware keyboard and trackpad support, decide whether direct left/right input should be honored or locked to vertical-drives-horizontal only.
  5. 05Test on mobile specifically, horizontal-pin sections are one of the most common places mobile scroll feels broken if not built carefully.
Prompt it
claude
Build a horizontal scrolling showcase section, the kind where normal vertical scrolling drives a row of project cards or case studies sliding across the screen while the section itself stays pinned. I'd default to GSAP ScrollTrigger for the pin and translate math since it gives the most precise control over exactly how far the track should travel relative to scroll distance, but I'm open to Locomotive Scroll or even a React Three Fiber based scene if you think a more spatial, actually-3D version of this would look more distinctive than a flat translated row, use your judgment on which approach fits the content best rather than defaulting to the simplest DOM version automatically. Whatever you build, make sure the pin distance is calculated from the actual content width rather than a guessed vh value, so there's no dead scroll space or abrupt cutoff at the end, and think about whether individual cards should have their own subtle secondary motion (a slight scale or parallax) as they cross the center of the viewport, rather than moving as one completely rigid block. I want this to feel like an actual directed sequence, not just a sideways scrollbar, so add whatever secondary polish you think earns its place, and make sure it degrades to a normal scrollable row on mobile rather than trying to force the pinned behavior on a touch device.
~/build/05-tilt-cards $

3D Tilt & Parallax Cards

Cards that subtly rotate in 3D space as the cursor moves across them, small effect, disproportionately large impact on how premium a site feels.

Resources
react-parallax-tilt
A drop-in React component for this exact effect.
npm install react-parallax-tilt
vanilla-tilt.js
Framework-agnostic version of the same idea.
npm install vanilla-tilt
Atropos
A dedicated library for multi-layer tilt cards with parallax sub-elements built in, worth using instead of hand rolling if you want depth layers inside the card itself.
npm install atropos
React Three Fiber
For when the card should be an actual 3D mesh rather than a flat element with a perspective transform.
npm install three @react-three/fiber @react-three/drei
How to build it
  1. 01Track the cursor's position relative to the card's bounding box, so the tilt direction is relative to where on the card the cursor sits.
  2. 02Map that relative position to a rotateX/rotateY range, keep it modest, large rotation reads as gimmicky rather than premium.
  3. 03Add a perspective value on the parent container, without it the rotation will look flat instead of dimensional.
  4. 04Layer a subtle glare or highlight element that moves opposite the tilt direction, this sells the 3D illusion far more than the rotation alone.
  5. 05Reset smoothly to a flat resting state on mouse leave, don't snap it back instantly.
Prompt it
claude
Build a 3D tilt effect for my card grid that feels genuinely premium, not the fairly generic version of this effect you see on a lot of template sites where the whole card just rotates uniformly toward the cursor. I want actual depth inside each card, the image or thumbnail, the title, and any icon or badge should each shift by a slightly different amount as the card tilts, so it reads as multiple layers at different depths rather than one flat plane rotating. Consider Atropos if it gets you this layered result faster than hand-building it with Framer Motion or GSAP, but if you think a hand-rolled version using rotateX/rotateY plus a couple of transform-translated child layers will look better or integrate more cleanly with the rest of my component styling, go that route instead. A moving highlight or glare layer that shifts opposite the tilt direction is something I definitely want, since it's what actually sells the illusion of a real light source, but use your own judgment on its opacity, size, and blend mode rather than a fixed spec. I care about the easing on entry and especially on mouse leave, it should settle back to resting state smoothly, not snap, and I'd rather you take the time to get the physics feeling right than ship the most minimal version that technically works.

WebGL Background (Three.js / Shaders)

An ambient, slowly moving particle field or shader gradient behind your hero or footer, this is the effect that makes a site feel like it was built by people who actually do this for a living.

Resources
Three.js
The underlying library most of the WebGL ecosystem is built on top of.
npm install three
React Three Fiber + drei
The React-idiomatic way to use Three.js, drei adds a huge set of prebuilt helpers on top.
npm install @react-three/fiber @react-three/drei
Shadergradient
A no-code / low-code way to get an animated shader gradient background without writing GLSL yourself.
OGL
A much lighter alternative to Three.js if your effect is simple and you care about bundle size.
npm install ogl
Pixi.js
A 2D-focused WebGL renderer, sometimes a better fit than Three.js if the background is closer to layered particle/sprite work than a true 3D scene.
npm install pixi.js
How to build it
  1. 01Decide what the background is actually representing, a WebGL background with no conceptual tie to the brand reads as decoration, not design.
  2. 02Keep the geometry count and shader complexity reasonable, this runs on every visitor's device, not just yours, test on a mid-range phone.
  3. 03Render it behind your content using a fixed position canvas with pointer-events disabled so it never blocks clicks on the real UI.
  4. 04Tie a couple of shader uniforms to scroll position or mouse position so it feels alive rather than a looping video.
  5. 05Add a static image or CSS gradient fallback for devices that fail WebGL context creation.
Prompt it
claude
I want an ambient WebGL background behind my hero section, something that feels alive and dimensional rather than a static image, but I want you to actually design what it represents rather than defaulting to a generic particle field. Think about what would tie back to the actual subject or brand of the site, an abstract shader gradient, a sparse field of drifting geometric shapes, a distorted wireframe mesh reacting to mouse position, or something else entirely if you have a better idea, and pick the direction you think looks the most distinctive rather than the most common WebGL demo effect. Use React Three Fiber with drei if you're building an actual 3D scene, or look at Shadergradient or a hand-written GLSL shader if a pure 2D shader effect would look cleaner and perform better, and consider OGL or Pixi.js instead of full Three.js if the scene ends up simple enough that a lighter renderer would be a smarter technical choice. Whatever you build, tie at least one or two of its properties (color, speed, distortion amount) to scroll position or cursor movement so it responds to the user rather than just looping passively, and make sure there's a real fallback, a static gradient or image, for devices where WebGL context creation fails, don't let the whole hero break in that case. Performance matters here more than in almost any other section, so keep an eye on draw calls and geometry count as you build rather than only checking it at the end.
~/build/07-mouse-parallax $

Mouse Parallax

Multiple layers of content drifting at different speeds and directions as the cursor moves, creating real depth without needing a full 3D scene.

Resources
Framer Motion
useMotionValue and transform() mapping mouse position to several layers at different depth multipliers, the most common approach in a React stack.
npm install framer-motion
GSAP quickTo
Cleanest option if you have many layers moving at once and want tight control over each one's easing independently.
npm install gsap
Atropos
A dedicated multi-layer parallax library if you want depth layers configured declaratively rather than wiring up transforms by hand.
npm install atropos
React Three Fiber
Worth using for the deepest background layer specifically, so it can sit in real 3D space instead of a faked 2D offset.
npm install three @react-three/fiber @react-three/drei
How to build it
  1. 01Decide how many depth layers you actually need, more layers rarely reads as more depth past a handful.
  2. 02Assign each layer a different multiplier for how far it moves relative to raw mouse delta, background layers should move less, foreground layers more.
  3. 03Normalize mouse position to a relative range based on the viewport or container center, rather than raw pixel coordinates, so behavior stays consistent across screen sizes.
  4. 04Ease each layer's movement independently rather than 1:1 to the cursor, slight lag per layer sells the depth illusion.
  5. 05Cap the maximum offset per layer so foreground elements never drift off-frame at extreme cursor positions, especially on ultrawide monitors.
Prompt it
claude
Build a mouse parallax system for my hero (or another section you think would benefit from it more) where multiple layered elements drift at different depths in response to cursor movement, background elements shifting subtly, closer elements shifting noticeably more, so the whole section reads as having real spatial depth rather than being a flat page with one thing following the mouse. I'd lean toward Framer Motion's motion values and transforms for this since it tends to compose well with layered React components, but use GSAP's quickTo instead if you think it'll give tighter, more independent control per layer, and consider Atropos if a purpose-built library produces a cleaner result than wiring up the transforms by hand. If it fits the content, explore putting the deepest background layer in an actual React Three Fiber scene rather than faking depth with 2D transforms, real 3D space for at least one layer often reads as more convincing than every layer being a flat div with a different multiplier. I want you to use your own judgment on how many layers to use, how far each should be allowed to travel, and what easing curve each layer gets, optimize for this feeling genuinely dimensional and premium rather than the simplest implementation that technically moves things on mousemove, and make sure it degrades sensibly on touch devices where there's no persistent cursor to track.

Interactive Masking

A shape or area that reveals what's underneath it, an image, a video, a color field, based on cursor position or scroll, like a spotlight the visitor drags around to uncover content.

Resources
CSS clip-path + mix-blend-mode
Native and dependency-free for simpler mask shapes, a great starting point before reaching for WebGL.
GSAP MorphSVG / DrawSVG
For animating the mask shape itself, not just its position, part of GSAP's paid Club plugin set but genuinely worth it for this.
Three.js custom shader mask
The most control, a shader-driven mask that can respond to cursor position in real time with a soft, feathered edge.
npm install three
Pixi.js
A lighter-weight WebGL option than Three.js if the mask is a 2D compositing effect rather than a true 3D scene.
npm install pixi.js
Rive
Good option if you'd rather author the mask shape or its animation visually instead of hand-coding SVG paths.
How to build it
  1. 01Decide whether the mask needs to be a simple CSS shape or a custom path, simple shapes don't need a library at all.
  2. 02If using clip-path, animate its radius or offset directly with GSAP or Framer Motion rather than relying on CSS transitions alone, for consistent easing control across the rest of the site.
  3. 03For text-as-mask (revealing a video or image only inside the shape of a headline), use mix-blend-mode or an SVG mask element, rather than trying to fake it with background-clip alone.
  4. 04Tie the mask's position or size to cursor coordinates for a spotlight-style reveal, or to scroll progress for a scroll-driven reveal, decide which fits the section before building.
  5. 05Profile performance specifically, WebGL-based masks can get expensive if the masked area is large and animating every frame.
Prompt it
claude
Add an interactive masking effect somewhere on the site where I think it'll have the most impact, a shape or region that reveals an image, video, or color field underneath it based on cursor position, like a spotlight or a window the visitor drags around to uncover content, rather than seeing the content plainly right away. I want you to decide whether this is best built with a CSS clip-path and mix-blend-mode approach, an SVG mask, or a full WebGL shader-driven mask using Three.js or Pixi.js, base that decision on how large the masked area is and how much visual richness the effect needs, not just on whichever is easiest to wire up. If GSAP's MorphSVG or DrawSVG plugins would let the mask shape itself morph or animate in a way that looks better than just repositioning a static shape, use them. I specifically don't want this to read as an obvious circle following the mouse, so think about a soft, feathered edge on the mask boundary, a bit of lag or easing rather than a perfectly rigid 1:1 follow, and whether a secondary smaller mask or subtle blend layer would add richness worth the extra complexity. Prioritize this feeling fluid, high-end, and a little surprising over it being the most literal or simplest interpretation of 'reveal on hover', and make sure there's a sensible fallback for touch devices where there's no persistent cursor position to drive it.

Text Scramble / Split-Text Reveals

Headlines that reveal letter-by-letter, scramble into place, or split and stagger in on scroll. Used sparingly, on one or two key headlines, not every paragraph.

Resources
SplitType
Splits text into individual characters, words, or lines as separate DOM elements you can then animate.
npm install split-type
Splitting.js
A similar splitting utility with a slightly different API.
npm install splitting
GSAP ScrambleTextPlugin
Purpose-built for the scramble-into-place effect specifically, part of GSAP's paid Club plugin set.
Framer Motion staggered children
Good enough for a simple word-by-word or line-by-line reveal without needing a separate splitting library.
npm install framer-motion
How to build it
  1. 01Reserve this effect for headlines and key statements only, applying it to body copy or long paragraphs will slow down reading and feel gimmicky.
  2. 02Split the target text into the right unit (characters for scramble, words or lines for a stagger reveal).
  3. 03For a stagger reveal, animate opacity and a small translateY per unit, staggered apart for characters or slightly more apart for words.
  4. 04For a scramble effect, cycle each character through a few random characters before settling on the real one, keep the scramble duration short so it doesn't overstay its welcome.
  5. 05Make sure the text is still selectable and accessible to screen readers after splitting.
Prompt it
claude
Design a text reveal treatment for my key headlines, something more considered than a generic word-by-word fade, I want it to feel like a genuine signature moment on the page rather than a stock animation. Use SplitType or Splitting.js to break the text into the right unit, characters if you're going for a scramble-into-place effect, words or lines if it's more of a staggered reveal, and feel free to combine techniques across different headlines rather than applying the exact same treatment everywhere, for example a scramble on the hero headline using GSAP's ScrambleTextPlugin, paired with a simpler staggered word reveal on a secondary subheading using Framer Motion. I want you to make real decisions about timing and character set for any scramble effect (should it cycle through random letters, symbols, or something tied to the content itself), and about stagger delay and easing for any word or line based reveal, rather than reusing one fixed configuration everywhere. Whatever you build, make sure it's fast enough that it doesn't make the page feel slow to read, and double check that the underlying text remains fully accessible to screen readers and still selectable by the user after the DOM gets split apart.
~/build/10-hover-distortion $

Hover Distortion Effects

Images or thumbnails that ripple, warp, or displace slightly on hover, a signature move of high-end portfolio and agency sites.

Resources
Curtains.js
A WebGL library built specifically for image plane distortion driven by cursor position or a displacement map.
npm install curtainsjs
Shery.js
Purpose-built for this category of effect, ships with several hover-distortion and text-distortion presets out of the box.
npm install sheryjs
Three.js custom shader (displacement map)
The most control, requires writing your own GLSL fragment shader, worth it if prebuilt libraries don't match your exact vision.
npm install three
hover-effect (Robin Delaporte)
A smaller, focused library specifically for the classic image-to-image WebGL hover transition/distortion look.
npm install hover-effect
How to build it
  1. 01Decide which images actually deserve this treatment, applying it to every thumbnail on a page will overwhelm rather than impress.
  2. 02Render the target image as a WebGL plane rather than a plain img tag.
  3. 03Feed cursor position into the shader as a uniform, and use it to drive a displacement or ripple amount localized around the cursor's position over the image.
  4. 04Tune the displacement strength and falloff radius, too strong and the image becomes unreadable mid-hover, too weak and the effect goes unnoticed.
  5. 05Provide a plain, undistorted image fallback for devices without hover capability.
Prompt it
claude
Add a hover distortion effect to the thumbnail images in my project or gallery grid, but rather than the very common circular ripple that just follows the cursor, I want you to actually design a distortion behavior that feels specific to this content, consider a directional displacement based on cursor velocity rather than just position, a chromatic aberration or RGB channel split on hover, or a subtle image-to-image crossfade transition using something like the hover-effect library if that reads as more premium for this particular grid. Use Curtains.js or Shery.js if a purpose built library gets you most of the way there, or write a custom Three.js shader with a displacement map if you want full control over exactly how the distortion looks and behaves. I want real thought put into the strength and falloff radius of whatever distortion you choose, too strong and the image becomes unreadable mid-interaction, too subtle and nobody notices it happened, so tune it until it feels right rather than shipping the first value that technically works. Make sure there's a plain, undistorted static image fallback for touch-only devices, since there's no hover state to trigger any of this there, and lean toward restraint, this effect should be reserved for a handful of images that deserve the attention, not applied uniformly across every thumbnail on the page.

Build Order

The order below isn't arbitrary, each layer depends on the one before it existing first. Building out of order is the most common reason these effects end up fighting each other.

  1. 01Set up your Expo project and file-based routing skeleton first, every effect below assumes a working page structure to attach to.
  2. 02Build the hero section and its scroll-pinned video next, it's the first thing every visitor sees and the highest-leverage section to get right early.
  3. 03Add scroll-triggered reveals to your standard content sections once the hero is solid.
  4. 04Layer in a horizontal scroll section only if your content actually calls for it, a gallery or case-study list, not just for the sake of having one.
  5. 05Add 3D tilt cards and a WebGL background once your core layout is stable, these are visual polish layers, easier to place once the structure exists.
  6. 06Add mouse parallax once you have real content layers to give depth to, it needs actual foreground/background elements to be worth building.
  7. 07Add interactive masking to the one or two sections that most benefit from a reveal moment, don't spread it thin across the whole site.
  8. 08Add cursor-based animations after the rest of the UI exists, it's a global overlay and easiest to tune once you can see it against real content.
  9. 09Add text scramble or split-text reveals to your key headlines once you know which ones actually deserve the emphasis.
  10. 10Add hover distortion to specific images last, once your gallery or portfolio grid is finalized, don't build it against placeholder content.
